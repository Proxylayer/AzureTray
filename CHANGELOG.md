## [Unreleased]

## [0.2.2] — 2026-05-12

### Changed

- **Host permission audit + restructure.** `HostRequiredPermissions` split into two layered sets:
  - `Baseline` — `User.Read` only. What the runtime host actually exercises (sign-in identity, /me lookup, tenant display name resolution).
  - `AdminTools` — `Application.ReadWrite.All` + `DelegatedPermissionGrant.ReadWrite.All`. Only needed by the in-app admin features (Fix Permissions, Create App Registration, App Registration search). A user who never invokes those features doesn't need them functionally.
  - `All` = `Baseline ∪ AdminTools`, used by the aggregation that drives Create App Registration / Fix Permissions provisioning. No behavioural change vs. v0.2.1; the split makes intent legible. Plugins continue to declare their own runtime scopes via `ITrayPlugin.RequiredPermissions`.

### Fixed

- **Log Viewer redesign.** Replaced the `DataGrid` (which truncated "Information" to "Informatio", horizontally scrolled, and made selected rows unreadable) with a virtualized `ListBox` whose rows lead with a 3-char severity glyph (`ERR`/`WRN`/`INF`/`DBG`/`VRB`/`FTL`) and an accent-colored stripe. Messages wrap. Exception text renders inline under the message in muted monospace. Class label is right-aligned and click-to-filter. Dropped the date pickers (not useful for a 500-entry ring) and collapsed the two filter rows to one. New `LevelToGlyphConverter` + `NullToVisibilityConverter`.
- **Host crash on shutdown.** `TrayIcon.Dispose()` called `_services.GetService<IPluginLoader>()` to unsubscribe `PluginsChanged`, but the IServiceProvider is disposed *before* singleton `Dispose()` runs. Cached the loader at `Start()` time and use that reference in `Dispose()` — no more `ObjectDisposedException` at exit (the `[FTL] Host terminated unexpectedly` entries).
- **PIM activation 400 Bad Request.** Both `GraphPimClient.ActivateRoleAsync` and `ArmPimClient.ActivateRoleAsync` were sending `startDateTime = DateTimeOffset.UtcNow.ToString("o")`. By the time the request reached the server, that timestamp was a few hundred milliseconds in the past, and Graph/ARM 400 any schedule with a past start. Now passes `null` (omitted via `WhenWritingNull`) so the service computes the instant itself. Matches legacy `Azure.PIM.Tray` behaviour.
- **Graph error response bodies now visible.** `AppRegistrationGraphClient` previously used `EnsureSuccessStatusCode` which discarded the response body, leaving Fix Permissions / Create App Registration failures as opaque "403 Forbidden". Now uses `EnsureSuccessOrThrowWithBodyAsync` (same pattern already in the PIM clients) so the user sees `Authorization_RequestDenied: Insufficient privileges...` and similar Graph error codes in the log.

## [0.2.1] — 2026-05-12

### Added

- **Single-instance enforcement.** The host now holds a per-user named mutex (`Local\Proxylayer.AzureTray.SingleInstance`); launching `AzureTray.exe` while it's already running for the same user logs and exits cleanly instead of starting a second tray icon. Crash-safe via `AbandonedMutexException` — a kill-3 leaves the next launch able to acquire.
- **Update-available notification.** When `IUpdateService.CheckOnStartupAsync` detects a new release and finishes downloading it, the host surfaces a non-timing-out `ActionRequest` toast with a blue accent stripe and an "Update now" button. Dismissing keeps the option available via the Settings banner; clicking "Update now" applies via Velopack and restarts.
- **Update banner in Settings.** A clickable blue banner above the cards shows when an update is pending; click to apply.
- **Default Velopack feed URL** set to `https://github.com/Proxylayer/AzureTray/releases/latest/download` so installs pick up the latest published release automatically. Override to `""` to disable update checks.

### Changed

- **Plugin versions decoupled from the host.** `AzureTray.Plugin.Contracts`, `AzureTray.Plugin.PIM`, and `AzureTray.Plugin.LAPS` each declare their own `<Version>` in their csproj. Host tag-driven releases no longer republish plugin packages; a separate manual workflow ([`publish-plugins.yml`](.github/workflows/publish-plugins.yml)) handles plugin publishes. `--skip-duplicate` keeps re-runs idempotent.
- `AzureTray.Plugin.Contracts` bumped to `0.2.1` (new types added — see below). `AzureTray.Plugin.PIM` and `AzureTray.Plugin.LAPS` remain at `0.2.0`.

### Plugin contract surface (additive, no PluginApiVersion bump)

- New `NotificationSeverity` enum (`Info` / `Update` / `Warning` / `Error`) on `NotificationRequest` base record. Drives the notification's accent stripe color.
- New `ActionRequest(Title, Message, ActionLabel)` notification type for single-call-to-action prompts. Never auto-dismisses.
- New `ActionResult(bool ActionInvoked)` distinguishes "user clicked the action" from `DismissedResult`.

### Fixed

- **Log Viewer wouldn't open**: `LogViewerViewModel` ctor assigned `SelectedClassOption` *before* `EntriesView` was initialised, so the partial `OnSelectedClassOptionChanged` threw NRE inside the ctor and the window never appeared. Reordered the ctor to initialise the CollectionViewSource first.
- **PIM activation failures had no user feedback**: a failed `Activate role` click logged the error but said nothing to the user. The watcher now surfaces a red error toast with the underlying Graph/ARM error code + message, and an info toast on the happy path. Powered by a new `EnsureSuccessOrThrowWithBodyAsync` helper in both PIM HTTP clients that preserves the service's response body in the exception (the standard `EnsureSuccessStatusCode` discards it).
- **Clipboard writes from background threads**: `HostClipboard.SetText` was failing with `ThreadStateException: Current thread must be set to single thread apartment (STA)` when invoked from a thread-pool thread (e.g. LAPS copy-password from a watcher continuation). Now marshals to the WPF dispatcher's UI thread before touching `System.Windows.Clipboard`.

## [0.2.0] — 2026-05-11

### Changed

- **Architectural pivot:** PIM is now a plugin, not core. The host is a generic tray + plugin platform; core responsibilities are auth, plugin host, logging, notifications, update, management. PIM, LAPS and other features ship as plugins.
- `IPluginContext` expanded with `IPluginHttpClient Http`, `IReadOnlyList<PluginTenant> Tenants`, `string GraphScope`, `string ArmScope`. Plugins now have everything they need to do authenticated Graph/ARM work without depending on host internals.
- Removed the Graph PIM code that briefly lived in the core host (`IGraphPimClient`, `GraphPimClient`, `GraphClientBase`, `Dto/Graph/Entra*`, `ODataPage<T>`). `GraphMeClient` inlined its own minimal token-fetch + send, since it's the sole remaining caller in core.
- `PluginOptions.TrustMode` default switched to `AllowUnsigned` for the pre-release period (dev convenience). Document tightening to `RequireSigned` or `RequireTrustedPublisher` before distribution.

### Added

- New project `AzureTray.Plugin.PIM`: `PimPlugin : ITrayPlugin`, internal `GraphPimClient` consuming `IPluginContext.Http`, internal Entra DTOs. Post-build target deploys the DLL to `%LOCALAPPDATA%\AzureTray.Data\plugins\` for local iteration.
- New contract types in `AzureTray.Plugin.Contracts`: `IPluginHttpClient`, `PluginTenant`, `PluginHttpClientNames`.
- Host implementation `HostPluginHttpClient` adapts `IPluginHttpClient` to `IHttpClientFactory` + `ICredentialFactory` (per-tenant token fetch, named client lookup).
- Renamed `AzureTray.Plugin.azPIM` → `AzureTray.Plugin.PIM` (the `az` prefix duplicated the `AzureTray` host name).
- **Interactive notifications.** New contracts: `INotifier` (added to `IPluginContext`), `NotificationRequest` (abstract base + `YesNoRequest`, `ChoiceRequest` with optional free-text "Other", `TextInputRequest`), `NotificationResult` (`YesNoResult`, `ChoiceResult`, `TextInputResult`, `DismissedResult`).
- Host `NotificationService` renders requests as frameless, topmost WPF popups in the bottom-right of the working area. Multiple notifications **stack** — each new one takes the next free slot above the existing ones; freed slots are reused. Each notification's result completes a `TaskCompletionSource` so callers `await` the user's answer; cancellation closes the window and yields `DismissedResult`. If the WPF dispatcher isn't running yet, `ShowAsync` returns `DismissedResult` immediately and logs a warning rather than hanging.
- `NotificationViewModel` covers all three input modes including the cross-deselect between the choice list and the "Other" radio. Submit gating is per-mode (text length / selection presence).
- **App registration management — three focused services in `AzureTray.AppRegistration`.** Replaces the earlier monolithic `IAppRegistrationManager`. (1) `IAppRegistrationDiscovery` — `FindByDisplayNameAsync`, `SearchByDisplayNameAsync`. (2) `IAppRegistrationPermissions` — `CheckAsync` reports missing scopes / unconsented scopes; `EnsureAsync` uses **replace semantics** (matches the legacy `FixPermissionsAsync` behavior — PATCHes `requiredResourceAccess` to exactly the host + plugin scope set, prunes stale scopes, replaces per-resource `oauth2PermissionGrants` scope strings, and DELETEs grants for resources no longer required). (3) `IAppRegistrationProvisioning` — `CreateAsync` provisions a brand-new app registration end-to-end (`POST /applications` single-tenant + `POST /servicePrincipals` + WAM broker redirect URI + admin-consented scopes), plus `EnsureBrokerRedirectUriAsync` for retrofitting existing app registrations. Successful destructive operations require Global / Application Administrator authority.
- Plugins declare their required delegated scopes via the new `ITrayPlugin.RequiredPermissions` (defaulted to empty). The PIM plugin declares all six scopes the original Azure.PIM.Tray needed (`User.Read`, `RoleAssignmentSchedule.ReadWrite.Directory`, `RoleEligibilitySchedule.Read.Directory`, `PrivilegedAccess.ReadWrite.AzureAD`, `RoleManagement.Read.Directory`, ARM `user_impersonation`) with their well-known scope IDs baked in.
- **PIM plugin now drives interactive approvals.** Internal `PendingApprovalWatcher` (one per tenant from `context.Tenants`) polls Graph every 60s, dedupes by approval ID, and on every new pending request shows a `ChoiceRequest("Approve", "Reject")` followed by a `TextInputRequest` for justification. The decision routes into `IGraphPimClient.ReviewAsync` (renamed from `ApproveAsync`, now takes `ApprovalDecision { Approve, Deny }`). Dismissed/cancelled notifications leave the approval untouched for the next cycle; approvals that drop out of the Graph list (acted on elsewhere) are forgotten so a re-request notifies again. Watchers are owned by `PimPlugin` and stopped on `ShutdownAsync`.
- **ARM PIM client (`IArmPimClient` + `ArmPimClient`)** ported to the plugin. Covers `ListSubscriptionsAsync`, `ListPendingApprovalsAsync(scopes)`, `ListEligibleRolesAsync(principalId, scopes)`, `CheckApprovalRequiredAsync(scope, roleDefinitionId)`, `ActivateRoleAsync(scope, principalId, roleDefinitionId, linkedRoleEligibilityScheduleId, duration, justification)`, `ReviewAsync(scope, approvalId, decision, justification)`, `GetActivationStatusAsync(scope, requestId)`. API versions used: `subscriptions=2022-12-01`, authorization resources=`2020-10-01`, approval stages=`2021-01-01-preview`. Activation uses a client-generated GUID `requestId` per ARM convention. All calls flow through `IPluginContext.Http` with `PluginHttpClientNames.Arm` + `ctx.ArmScope`. ARM DTOs (subscriptions, schedule request + properties, expanded principal/role/scope, approval + stages, eligibility schedule, policy assignment + rules, status) live in `Arm/Dto/` as internal records.
- `PimPlugin` now constructs an `ArmPimClient` alongside the Graph client.
- **Log Viewer now a sortable / filterable `DataGrid`.** Columns reordered to **Timestamp · Type · Message · Class**. Clicking any column header sorts ascending / descending. Each header carries its own filter `TextBox` (case-insensitive contains-match); filters AND together. View is backed by `CollectionViewSource` wrapping `ObservableCollection<LogEntry>`; the four filter properties (`TimestampFilter` / `TypeFilter` / `MessageFilter` / `ClassFilter`) refresh the view on change. New "Clear filters" toolbar button alongside "Clear log". Auto-scroll follows the filtered+sorted view's tail. Message column wraps; Class column trims with tooltip.
- **HTTP debug logging at the plugin chokepoint.** `HostPluginHttpClient.SendAsync` now traces every outgoing plugin request — `→ <client> <method> <url> (tenant <id>)` and `← <client> <method> <url> <status> (<elapsed-ms> ms, tenant <id>)`. Successful responses log at Debug; 4xx/5xx and thrown exceptions log at Warning with the elapsed time. Flip the Log Viewer's Capture level to Debug to see the full Graph/ARM call stream — fastest way to diagnose missing eligible roles, auth failures, etc.
- **Log Viewer window.** New WPF window bound to `LogRingBuffer` shows all in-memory log entries with timestamp, level, category, and message — level color-coded via `LevelToBrushConverter`. The "Capture level" dropdown is two-way-bound to `LoggingLevelSwitch.MinimumLevel`, so flipping to Debug or Verbose enables more detailed events on the fly (essential when troubleshooting eligible-role polling). Auto-scroll checkbox follows the tail; Clear button wipes both the buffer and the bound collection. New `LogRingBuffer.ErrorCount` / `WarningCount` counters update on every Add/eviction; `Clear()` resets them. Tray context menu gains "📝  Log Viewer" between Settings and Quit, with a `(✕ N)` / `(⚠ N)` badge sourced from those counters. Window can be opened multiple times (lifecycle-managed by `TrayIcon` like the Settings window).
- **Active-role grayout.** `EligibleRolesWatcher` now also fetches `IGraphPimClient.ListActiveRoleAssignmentsAsync` each poll and exposes `CurrentActiveRoleNames` (case-insensitive `IReadOnlySet<string>`). `PimPlugin.AppendSourceGroup` checks this set when rendering each eligible role: matches show as disabled `"<Role>  (<Scope>)  ✓ active"` (no click), non-matches stay clickable. Same name set is applied to both Entra and ARM eligible roles, matching the predecessor's loose check.
- **"Open Request" menu + eligible-role activation.** New `EligibleRolesWatcher` per tenant (30-min cadence to mirror the predecessor; click "↻ \<Tenant\>" to force refresh). Resolves and caches the signed-in user's principal ID on first poll, then fans out `IGraphPimClient.ListEligibleRolesAsync` + `IArmPimClient.ListSubscriptionsAsync` → `ListEligibleRolesAsync(scopes)` and projects results into a source-agnostic `UnifiedEligibleRole` snapshot. `PimPlugin.GetMenuItems()` now contributes a second top-level item "🔑 Open Request (n role(s))"; submenu lists each tenant with its eligible roles grouped under disabled "— Entra ID —" / "— Azure RBAC —" headers and ordered by role name. Clicking a role fires `HandleActivationAsync`: `ChoiceRequest` for duration (1h / 4h / 8h) → `TextInputRequest` for justification → routes to `IGraphPimClient.ActivateRoleAsync` (Entra) or `IArmPimClient.ActivateRoleAsync(scope, principalId, roleDef, eligibilityId, duration, justification)` (ARM). Dismiss at either prompt or a blank justification aborts cleanly.
- **PIM menu matches the original layout.** Reshaped `PimPlugin.GetMenuItems()` to mirror `Azure.PIM.Tray`'s `ContextMenuBuilder`: a single top-level "⏳  Pending Approvals  (n)" submenu, with each managed tenant appearing as a clickable "↻  \<Tenant\>" refresh header followed by the actual pending requests inline (`<Principal> — <Role>`), or a disabled "(none)" placeholder. Clicking a request invokes the same Approve/Reject + justification flow used by auto-discovery, routed back to Graph or ARM by source. Tenants are visually separated by `PluginMenuItem.Separator`. `PendingApprovalWatcher` now exposes `CurrentApprovals` (snapshot of the most recent poll) so the menu can render the live state without re-polling on every right-click. `PluginMenuItem` gains an `IsSeparator` flag + `PluginMenuItem.Separator` singleton, and `TrayIcon` renders separators as `ToolStripSeparator`s. **Note:** "Open Request" (eligible role activation) is not yet ported — it needs eligible-role tracking added to the watcher, which is the next piece.
- **Plugin tray menu contributions.** New contract piece: `PluginMenuItem` record + `ITrayPlugin.GetMenuItems()` (default-implemented to return an empty list, so existing plugins don't change). The host's `TrayIcon` rebuilds its `ContextMenuStrip` on every `Opening` event — plugin items first (in load order, exception-isolated per plugin), separator, then the existing Settings / Quit. Submenu nesting via `PluginMenuItem.Children`; disabled headers via `IsEnabled = false`. PIM plugin contributes an "Azure PIM" submenu with a "Refresh \<tenant\>" entry per active watcher; clicking forces an immediate `PendingApprovalWatcher.PollAsync` so users can pull pending approvals on demand without waiting for the 60 s cycle. `PendingApprovalWatcher` now publicly exposes `TenantId` and `TenantDisplayName`.
- **Extension installer UI.** Settings window now has an "Extensions" panel listing every plugin DLL in the plugins folder, with **Install from file...** and **Open plugins folder** buttons and an Uninstall button per row. Behind it:
  - `IFileDialogService` (`Shell/`) wraps `Microsoft.Win32.OpenFileDialog` for testability.
  - `IExtensionInstaller` (`Extensions/`) handles `InstallFromFileAsync` (streamed copy with overwrite), `RequestUninstallAsync` (writes a `.uninstall` sentinel next to the DLL — the running process holds the DLL open, so the actual delete can't happen until restart), `ListInstalledDlls` / `ListPendingUninstalls`, and `OpenPluginsFolder`.
  - `PluginLoader.LoadAllAsync` now calls `ProcessPendingUninstalls()` before scanning, so the deletion completes on the next launch.
  - `InstalledExtension` view model record joins file-on-disk with `IPluginLoader.LoadedPlugins` so the UI shows display name + version when loaded, "Installed — restart to load." when not, and "Pending uninstall — restart to apply." when sentinel-flagged.
- **Add-tenant escape hatch.** Three fixes so a bad client ID can't trap the Settings window:
  - `Client ID` field is validated as a GUID before any persistence or sign-in. Non-GUID values show an inline error; the store and Graph are never touched.
  - New `CancelAddTenantCommand` cancels an in-flight sign-in. The Cancel button sits next to "Add & Sign In" and is `CanExecute`-gated on `IsAddingTenant`. Cancellation rolls back the tentative store entry and invalidates the cached credential.
  - Hard 90-second sign-in timeout (`CancellationTokenSource.CancelAfter`) so even a hung `InteractiveBrowserCredential` releases the UI on its own.
- **Watcher unified across Graph + ARM.** `PendingApprovalWatcher` now polls both sources per cycle in parallel, projects each pending request into a source-agnostic `UnifiedPendingApproval(Source, ApprovalId, PrincipalDisplay, RoleDisplay, ScopeDisplay, ArmScope?)`, dedupes on `(Source, ApprovalId)`, and routes the user's Approve/Reject decision back to the originating client — Graph approvals to `IGraphPimClient.ReviewAsync`, ARM approvals to `IArmPimClient.ReviewAsync(scope, ...)` using the scope captured from the schedule request. A failure in one source (e.g. ARM down) is caught and logged; the other source is still processed.
- **Three add-tenant modes.** Settings's primary split-button drives a single command whose action depends on the selected mode: **Windows account** (auto-detect via OIDC on the signed-in UPN's domain, fallback to broker picker), **Email** (always opens the WAM picker — type any email or pick another work/school account configured in Windows Settings → Accounts → Email & accounts), or **Manual** (typed Tenant ID + optional Client ID + optional app-registration name lookup). Picking a sign-in mode from the dropdown immediately fires the action — no extra button click required. Picking the already-active mode is suppressed with a passive notification ("Already in this mode") to avoid accidental duplicate broker prompts.
- **Tenant edit.** `[Edit]` button on each tenant row pre-fills the manual form with the tenant's display name + client id and locks the Tenant ID field. Save changes persists in place, preserves `SignInEmail` / `ProbeDisabled`, invalidates the cached credential, and refreshes the plugin per-tenant config rows. A `Cancel edit` button bails out without changing anything.
- **Plugin per-tenant config refreshes on add/remove.** `SettingsViewModel.RefreshPluginConfigs()` fires after both add paths and after `RemoveTenantAsync`, so the per-plugin tenant toggle rows appear / disappear without reopening Settings.
- **Manual setup auto-collapses on save.** After a successful Save tenant (manual) or Save changes (edit), the form folds back to Windows-mode default. Eliminates the always-expanded form lingering after a successful add.

### Removed

- **`BearerTokenHandler`.** Replaced everywhere by `ICredentialFactory.GetForTenant(...)` + per-call `GetTokenAsync` (or `HostPluginHttpClient` on the plugin side). The delegating-handler form was never wired into a request pipeline after the auth refactor.

### Added

- **App-registration end-to-end provisioning + replace-semantics fix.** See the architectural bullet above. New result types `AppRegistrationCreateResult(App, ScopesGranted, BrokerRedirectUriAdded)` and the enriched `PermissionFixResult(ScopesAdded, GrantsAdded, StaleScopesRemoved, StaleGrantsRemoved)`. Host-side baseline scopes live in `HostRequiredPermissions` (currently `User.Read`); plugin scopes come from `ITrayPlugin.RequiredPermissions`; `SettingsViewModel.AggregateRequiredPermissions()` dedupes by `(Api, ScopeId)` before calling Ensure/Create.
- **Admin Settings mode.** Tray's admin menu now has a "⚙ Settings (admin)…" entry separate from the regular menu's "⚙ Settings…". Admin mode is reflected in the window title ("Settings — Administrator") and unlocks per-row **Fix permissions** and **✚ App reg** buttons on each tenant row. User-mode Settings hides those buttons. The tray icon manages window mode-switching — opening admin Settings while user Settings is open closes the latter and vice versa.
- **Contextual "Create app registration" prompt.** When the Add Tenant flow finishes without binding a tenant to a dedicated app registration (no `App:Auth:AppRegistrationName` match in the tenant), the Add Tenant panel shows a follow-up "✚ Create app registration in {tenant}" button so the user can provision one in one click. Dismissed once Create succeeds or the user adds a different tenant.
- **`Tenant` sign-in metadata.** Two new optional fields on the `Tenant` record:
  - `SignInEmail` — the UPN the tenant was originally signed in with. Used as MSAL's `LoginHint` so the broker pre-fills the right account when an interactive sign-in is needed later. Populated from the WAM account on Windows-mode add, from Graph `/me` on manual save.
  - `ProbeDisabled` — when true, the startup readiness probe skips this tenant and never auto-prompts. Set by the "Disable" button on the startup sign-in notification; cleared by any successful interactive sign-in.
  - Both fields are optional with sensible defaults — older `tenants.json` files written by the 3-field shape deserialize cleanly.
- **Startup readiness rework: silent-by-default with a Login/Disable notification on cache miss.** `CredentialFactory.GetForTenant` now builds the silent credential with `DisableAutomaticAuthentication = true` + `UseDefaultBrokerAccount = true` so cache hits and the broker's signed-in-Windows-account path succeed silently, and anything else throws `AuthenticationRequiredException` without popping a window. The probe catches that, stacks a non-timing-out `ChoiceRequest` notification ("Sign in to {tenant}…  [Login] [Disable]"). **Login** drives the broker via `ICredentialFactory.SignInAsync` (runs on the WPF dispatcher with a real top-level window HWND so WAM anchors correctly); **Disable** persists `ProbeDisabled = true`. Sign-in failures surface a follow-up notification so the user is never left guessing.
- **`AuthenticationRecord` persistence.** `SignInAsync` writes the post-broker `AuthenticationRecord` to `%LOCALAPPDATA%\AzureTray.Data\auth-records\{tenantId}.bin` and `BuildForTenant` reads it on first miss. The record is a non-secret pointer (UPN + home-account-id + tenant + clientId) that tells MSAL which cached account to look up; the actual tokens live in the MSAL persistent cache and are DPAPI-encrypted. With both files present the startup probe can use the cached refresh token silently, restart after restart.
- **Disabled-tenant row styling + Sign in button.** Tenants with `ProbeDisabled = true` show dimmed text and an italic "(disabled)" badge; the per-row action buttons are replaced by `[Sign in] [Remove]`. Clicking Sign in invokes `ICredentialFactory.SignInAsync`, persists `ProbeDisabled = false` on success, marks the tenant ready, and re-renders the row in its enabled state.
- **Admin Test Runner: "Host: App Registration" group.** Five tests against the live Microsoft Graph: `Discover by display name` (read-only), `Check permissions on current app reg` (read-only), `Ensure WAM broker redirect URI` (idempotent), `Fix Permissions` (DESTRUCTIVE, Yes/No gated), `Create App Registration` (DESTRUCTIVE, Yes/No gated, persists the new clientId on the chosen tenant). Tenant picker dialog filters to eligible tenants per test.
- **Log Viewer filters redesigned.** Replaces the four substring text boxes with: **From** + **To** `DatePicker` + optional `HH:mm:ss` time text boxes (empty = start/end of day), **Type** `ComboBox` over the level enum + "(All types)", **Class** `ComboBox` grouped by namespace root (e.g. `AzureTray.Auth`, `Microsoft.Extensions`) with "(All classes)" sentinel; categories grow dynamically as new entries arrive in the buffer, **Search** text box (case-insensitive contains-match against the Message column). Selected-row text now uses the primary foreground brush so it's readable on the dark accent background.
- **Tray menu scroll arrows.** When the menu height exceeds `MaxHeight=500`, ▲/▼ borders appear at the top/bottom edges (ported from legacy `Azure.PIM.Tray`). Hovering an arrow auto-scrolls one row per 300 ms via a `DispatcherTimer`; mouse leave stops the timer. Visibility is driven from the inner `ScrollViewer`'s `ScrollChanged` event (located via visual tree on `ItemsList.Loaded`). The main menu now opens with the cursor 3 DIP inside the bottom-right corner so subpixel rounding can't dismiss it.
- **Plugin loading is no longer on the critical startup path.** `PluginLoader.StartAsync` returns `Task.CompletedTask` immediately and queues `LoadAllAsync` on the thread pool, so `host.Start()` doesn't block on plugin assembly JIT (was 5-10+ s with multiple plugins). The tray icon appears in ~1-2 s; plugins announce themselves via `PluginsChanged` as each finishes loading.
- **Add-Tenant status: red error styling + proactive interactive sign-in.** `AddTenantStatusIsError` flag colors failure messages with `Brush.Status.Error`. The manual Save tenant path now proactively calls `ICredentialFactory.SignInAsync` before any Graph call so the per-tenant MSAL cache is populated before `/me` runs (was failing with "Interactive authentication is needed").
- **Build memory: kill running instance + Release-only.** Two persistent build preferences recorded in the project memory: before any `dotnet build` / `publish`, stop any running `AzureTray.exe` (silent on no-instance) so file locks don't break MSBuild; always pass `--configuration Release`.

- Release pipeline (`.github/workflows/release.yml`): build + test + Velopack self-contained `win-x64` package + CycloneDX SBOM + GitHub build-provenance attestations + draft GitHub Release. Triggered by `vX.Y.Z` tag push or manual dispatch with a version input. Code-signing step is wired in as a commented placeholder (uncomment and supply `SIGN_CERT_PFX_BASE64` / `SIGN_CERT_PASSWORD` secrets to enable).
- `RepositoryUrl`, `RepositoryType`, `PackageProjectUrl`, `Description` set in `Directory.Build.props` so packaged binaries carry consistent metadata.
- New project `AzureTray.Plugin.Contracts` (targets `net8.0`, NuGet-packable) with `ITrayPlugin`, `IPluginContext`, and `PluginApiVersion.Current = 1`. `AssemblyVersion` pinned to `1.0.0.0` so plugins continue to bind across host releases until the contract breaks.
- Host-side plugin loader: per-plugin collectible `AssemblyLoadContext` (shares host-loaded assemblies for contract/logging types, plugin-private for the rest), `ITrayPlugin` discovery, `ApiVersion` gate, lifecycle (`InitializeAsync`/`ShutdownAsync`), exception-isolated load and shutdown.
- `IPluginSignatureVerifier` + `AuthenticodePluginSignatureVerifier` (extracts Authenticode signer cert thumbprint; chain validation deferred).
- `PluginOptions` with three trust modes: `AllowUnsigned` (dev only), `RequireSigned` (default), `RequireTrustedPublisher` (thumbprint allowlist).
- `PluginLoader` registered as a singleton `IHostedService`; plugins are loaded on host start and unloaded on host stop.
- First feature port: sign in to a tenant from the Settings window. Adds `IGraphMeClient` (`GET /v1.0/me`) composed over `IHttpClientFactory` (named `graph` client with resilience handler) + `ICredentialFactory` + `IAzureCloudConfig.GraphScope`. Settings ViewModel exposes `TenantId`, `SignInCommand`, and `SignedInAs`. First end-to-end exercise of the auth + HTTP stack the previous steps built.
- First DTO in `AzureTray.Dto/` (`MeResponse`), demonstrating the wire-shape convention.
- Multi-tenant support: `Tenant` model, `ITenantStore` + `JsonFileTenantStore` persisted to `%APPDATA%\AzureTray\tenants.json` (atomic write via tmp-rename), `IAppPaths.TenantStoreFilePath`. `CredentialFactory` resolves per-tenant `ClientId` from the store with `App:Auth:ClientId` as fallback, and exposes `Invalidate(tenantId)` so the cache stays in sync with store edits. Settings UI lists configured tenants and includes an "Add tenant" form (Tenant ID / Display name / Client ID); successful sign-in via Graph `/me` persists; failure rolls back the store entry.

## [0.1.0] — initial refactor foundation

### Added

- Generic host + DI container boots before WPF; every service resolved from the container.
- Serilog logging with rolling file sink, debug sink, in-memory `LogRingBuffer` sink, and runtime `LoggingLevelSwitch`.
- `IConfiguration` layered from shipped `appsettings.json` + `%APPDATA%\AzureTray\config.json` + environment variables; typed `*Options` bindings (`UpdateFeedOptions`, `LoggingOptions`, `AzureCloudOptions`, `AuthOptions`).
- `IAppPaths` service. All user data stored at `%LOCALAPPDATA%\AzureTray.Data\` (logs, plugins) and `%APPDATA%\AzureTray\` (config) — deliberately outside the Velopack-managed install dir so updates never wipe them.
- `IAzureCloudConfig` centralising Graph, ARM, and Authority endpoints with sovereign-cloud overrides.
- `IHttpClientFactory` with named clients (`graph`, `arm`) wired through the standard resilience handler (retry, circuit breaker, timeout, 429 backoff).
- `ICredentialFactory` builds `InteractiveBrowserCredential` per tenant with MSAL persistent token cache, wrapped in per-tenant `SerializedTokenCredential` with configurable acquisition timeout.
- `CommunityToolkit.Mvvm` adopted; `SettingsWindow` ported as the MVVM template.
- `IUpdateService` interface extracted from `UpdateService`; settings VM depends on the interface.
- Test project `AzureTray.Tests` (xUnit + NSubstitute) with 18 tests covering `AppPaths`, `AzureCloudConfig`, `SerializedTokenCredential`, `SettingsViewModel`, and `LogRingBuffer`.
- GitHub Actions `pr.yml` workflow gating every PR on a Windows build + test.
- `Directory.Build.props`, `Directory.Packages.props`, `.editorconfig`, `nuget.config` â€” centralised build properties, central package management, code style, locked package source.

### Notes

- This release is foundation only; no features from `Azure.PIM.Tray` have been ported yet.
